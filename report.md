# 期末综合项目报告：AI多语言翻译润色Web工具
## 一、选题与需求分析（必做模块）
### 1. 项目解决的实际问题
市面上通用翻译工具仅提供基础直译功能，无法区分学术论文、日常口语、精简总结等多场景输出；学生撰写外文作业、英文论文时，缺少一站式翻译+专业润色+词汇解析一体化工具。
### 2. 目标使用人群
本工具面向在校大学生、课程论文创作者，满足日常外文阅读、学术写作、短句翻译的需求。
### 3. 项目核心价值
搭建轻量化本地Web网页，实现中英日韩四语种互译，同时提供4套不同风格润色方案，自动解析文本内专业生词，降低外文写作学习成本。
### 4. AI在项目中的角色与选型理由
1. AI承担工作：多语言语义转换、多风格文本改写、专业词汇提取释义；
2. 使用模型：智谱大模型 glm-4-flash 免费云API；
3. 选择AI而非固定规则的原因：
    传统硬编码规则翻译只能完成字面词语替换，无法理解上下文语义，也不能灵活切换学术/口语等多种写作风格；大模型具备自然语言理解与生成能力，可根据指令动态调整文本表达风格，适配多场景翻译润色需求。

## 二、系统架构与技术选型（必做模块）
### 1. 三层系统架构与数据流
1. 前端交互层：HTML+Bootstrap页面，提供输入框、语言选择下拉、功能按钮，通过JS异步向后端发送翻译请求，展示模型返回结果；
2. 后端服务层：Python Flask框架，接收前端JSON请求，封装翻译Prompt指令，调用大模型API，将返回文本传回前端；
3. 大模型推理层：智谱云端glm-4-flash免费模型，接收后端传入的翻译指令与原文，完成翻译润色计算后返回文本。
数据流：用户输入文本 → 前端发送POST请求 → Flask后端读取参数、拼接Prompt → 调用智谱API获取结果 → 数据传回前端页面渲染展示。

### 2. 完整技术选型
1. 编程语言：Python 3
2. Web后端框架：Flask
3. 前端技术：原生HTML、Bootstrap5、原生JavaScript
4. AI实现路线：调用大模型云API（智谱glm-4-flash，永久免费）
5. Python依赖库：flask、python-dotenv、langchain-openai
6. 硬件：普通Windows笔记本，无额外硬件设备

## 三、关键实现说明（必做模块）
### 1. 环境变量密钥隔离
使用`.env`文件单独存放智谱API密钥，通过`python-dotenv`库读取，密钥不硬编码写入代码，上传GitHub时不提交.env文件，避免隐私密钥泄露，符合作业禁止项规范。

### 2. 模块化Prompt封装
后端根据前端下拉框选中的润色模式，动态拼接不同指令的Prompt模板，区分标准翻译、学术润色、口语简化、精简压缩4种逻辑，统一规定输出三段式固定格式，保证返回文本排版整洁。

### 3. 接口健壮性处理
1. 输入空文本拦截：后端检测输入为空时，直接返回提示文本，不发起无效API请求；前端同步增加空白输入校验，提前拦截无效操作；
2. 接口超时限制：设置大模型调用timeout=15秒，防止网络异常时程序无限卡死；
3. 输出长度限制：配置max_tokens=1000，限制单次返回文本最大长度，缩短免费API推理等待时间，缓解响应慢问题。

### 4. 前端交互体验优化
新增加载等待提示，点击翻译按钮立刻展示“翻译中，请稍候”文字，解决用户误以为页面卡死无响应的问题；使用pre-wrap样式保留大模型返回文本换行，阅读排版清晰；增加一键清空按钮，快速重置输入输出区域。

## 四、测试结果与局限性分析（必做模块）
### 1. 有效测试场景
1. 英文专业论文句子 → 中文+学术润色：输出严谨书面化译文，自动解析文中专业名词；
2. 中文日常短句 → 日语口语简化：生成自然生活化短句，适配聊天场景；
3. 中文长段落 → 英文精简压缩：保留原文核心信息，缩短冗余语句；
4. 空白文本输入：前后端双重拦截，弹出提示，不会发起无效请求。

### 2. 项目局限性（缺点，作业必须写）
1. 依赖公网免费API，晚间使用高峰期服务器排队，页面响应速度较慢；
2. 小众冷门专业领域专有名词翻译准确度不足，无专业知识库支撑；
3. 单次文本输出长度受max_tokens参数限制，超长篇文档无法一次性完整处理；
4. 仅支持中英日韩四种语言，未覆盖俄语、德语等小语种。
